# EventSpace 高保真原型覆盖清单

> 本清单用于完成 Phase 0 的审美、排版和关键交互验收。所有页面均为静态高保真原型：使用本地示例内容和轻量交互，不连接 Supabase、不执行真实登录、支付、上传、推送或房间权限操作。

> 说明：`/prototype` 中旧 Room 横向滑动方案仅保留为历史视觉原型。正式 `/rooms/[roomId]` 已按后续决定改为“返回按钮 + 房间名下方常驻三项导航 + 禁用滑动切换”，正式实现与产品规格优先。

> 2026-07-14 更新：`/prototype`、`/prototype/flows`、`/prototype/operations` 只保留视觉历史。当前正式可操作 Mock 的覆盖与安全边界以 `docs/interactive-mock-coverage.md` 为准；不能再将静态原型计为功能完成。

## 1. 原型入口

| 原型批次 | 地址 | 已覆盖页面与状态 |
| --- | --- | --- |
| 核心产品 | `/prototype` | Landing、Rooms、Room；Room 内含 Chat、Board、Itinerary 连续横向轨道 |
| 进入与留存流程 | `/prototype/flows` | 分步创建、邀请与访客身份、归档入口、归档中/访问终止/配额状态、照片顺序流与详情 |
| 运营与管理流程 | `/prototype/operations` | 登录、入房审批、邀请分享、成员/举报、投票、行程详情、房间设置、房间选项、账户、法律页 |

原型顶部的深色切换条只用于评审页面与明暗主题，不属于正式产品导航。

## 2. 核心产品原型

### Landing

- 未登录访客看到的独立营销入口。
- 只有创建、通过邀请进入、登录和法律入口；没有公开房间或活动发现流。
- 使用抽象图钉照片与 UI 片段，不绑定活动主题，不使用渐变、纹理或高斯模糊。

### Rooms

- 登录后的产品首页直接显示 `Active / Archived` 与房间卡片。
- 移动端默认单张横向杂志式浏览，可切换双列纵向浏览。
- 卡片以完整画板缩略图为主要内容，信息区显示名称、状态、时间和人数。

### Room

- 顶部无返回按钮；左侧仅显示当前页图标，房间名严格居中。
- Chat、Board、Itinerary 同时保留在连续横向轨道中，拖动时跟手，松手后吸附。
- 当前页图标可临时展开其他页面图标，但没有常驻三标签栏。
- 桌面端保留低干扰房间侧栏；移动端以主内容优先。

## 3. 进入与留存流程原型

### 创建与进入

- 创建房间采用逐步引导，不使用一次性长表单。
- 展示 Host-led / Community-led 不可切换的模式选择。
- 邀请页在加入前仅展示房间名、说明、剩余时间和人数上限，不泄露成员或房间内容。
- 访客必须先设置昵称和头像。

### 归档与照片

- 归档入口突出完整画板、日期、参与人数和照片数量。
- 归档中、访问终止和免费照片配额达到上限均有独立状态表达。
- Sequence 使用稳定阅读顺序的两列纵向图片流；移动端详情使用底部面板，桌面端使用右侧详情栏。
- 详情包含发布者、注释、表情回应、评论与保存照片的视觉位置。

## 4. 运营与管理流程原型

### 身份与访问

- 登录：Google 与无密码邮箱链接；包含 16 岁与条款/隐私确认。
- 访客升级登录时明确继承当前昵称、头像和房间记录。
- 审批：Host/管理员看到头像、昵称和申请备注；同时提供申请者等待状态。
- 分享：链接、邀请码、二维码和复制入口；链接/二维码刷新与邀请码作废相互独立，二维码只是当前链接的可视化。

### 房间治理

- 成员：Host/Admin/Member/Guest 身份、管理员设置、禁言、踢出、设备级阻止再次进入。
- 举报：只提供举报位置描述和 Host 私密回复，不创建公开处理记录。
- 投票：赞成/反对或单选结构、公开/匿名、截止时间、固定分母说明和始终可见的票数。
- 行程详情：时间、地点、说明、负责人、三种状态、参与状态、人数上限、提醒与外部地图入口。
- 房间设置：入口审批、邀请码、结束时间、人数、照片容量、时区、通知、Host 转让与结束房间。

### 账户、付费与法律

- 房间选项：活动扩展与永久归档分开；均为房间级一次性权益，没有个人订阅或站内分摊。
- 原型不展示最终价格，避免在商业参数未定时形成错误承诺。
- 账户：昵称/头像、主题、个人归档列表移除、登录与隐私入口、退出和删号。
- 法律：服务条款、隐私政策、社区规则、Cookie 说明与支持入口；托管加密表述不宣称严格 E2EE。

## 5. 本轮不单独制作的状态

以下内容会在正式 mock 数据阶段作为既有页面的状态或底部面板实现，不需要为审美验收复制成大量独立页面：

- 每一种确认弹窗、Toast、加载骨架和网络重试文案。
- Chat 的每一种消息菜单、搜索结果、回复/撤回/置顶细节。
- Board 上传、拖动、遮挡检测和自动吸附的每个中间帧。
- Host 与 Community-led 权限组合下的每一种按钮禁用状态。
- 支付失败、Webhook 重试、归档清理和数据主体请求的后台过程。

这些状态仍属于正式功能范围，并将在权限矩阵、mock 场景和自动化测试中覆盖。

## 6. Phase 0 验收门槛

进入正式 mock 数据开发前，应确认：

1. 三个原型入口的整体设计语言、排版密度和移动端结构没有方向性异议。
2. Landing、Rooms 和 Room 的产品层级正确。
3. Room 的连续横向切换、画板视觉和行程信息层级符合预期。
4. 登录、邀请、审批、治理、付费与法律页面的语气和操作层级可以作为正式开发依据。
5. 允许后续在不改变产品规则的前提下，针对真实数据长度、Safari 行为和性能测试调整局部尺寸与动效参数。

验收后，原型目录冻结为视觉参考；正式实现按路由、feature、共享 UI、数据访问和权限层重新拆分，并首先使用类型化 mock 数据完成完整前端流程。
