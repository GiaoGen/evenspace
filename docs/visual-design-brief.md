# EventSpace 视觉设计简报（已确认部分）

> 状态：Phase 0 的页面方向与交互层级已确认。下一步为设计 token、线框/高保真稿与可复核交互原型；尚不进入功能实现。

## 1. 总体方向

- 产品应呈现为一个强沉浸、私密而高级的活动空间，而非传统 SaaS 后台或冗长营销站。
- 首屏以情绪化英文标题传达价值，例如 `Make the moment live longer.`；功能说明退居第二层。
- 视觉关键词：高级、轻盈、克制、流畅、活动感。
- 不提供按婚礼、派对、酒吧等场景区分的主题或模板。

## 2. 色彩、材质与字体

- 明亮与深色模式同等完整，不把任一模式降级为附属主题；默认跟随系统，用户可手动固定。
- 采用柔和高级的暖灰、纸白感的浅色底与墨黑，而不是纯黑纯白的硬对比。
- 禁止使用渐变色；不使用纸张、软木或木纹等拟物纹理。
- 依靠卡片底色、细边框和柔和阴影建立层级，避免高斯模糊/毛玻璃等高性能成本效果。
- 使用“高对比衬线标题 + 精致无衬线正文”的杂志式排版。最终字体家族、字重和授权仍待选择。

## 3. 形状、卡片与图像语言

- 使用 iOS 感的大圆角（约 20–28px）；卡片带可感知但不厚重的柔和阴影。
- 首页与归档卡片以单张“杂志封面”式浏览为主要体验；卡片画板缩略图约占 75%，其余区域展示房间名称、状态、结束时间、成员数。
- 首页营销主视觉使用抽象图钉照片语言和 UI 模拟图，而非真实活动人像或特定场景照片。
- 画板照片延续已确认的固定微倾斜、顶部小白条和中央图钉，不用渐变或强特效。

## 4. 信息架构与响应式排版

- 公开首页：情绪化首屏、主按钮 `Create a room`、次按钮 `Join with a link`、右上角登录；下方仅以三张精致卡片介绍 Chat、Board、Itinerary，不做冗长营销叙事。
- 登录后的首页：移动端默认以一次展示一张卡片的横向滑动流呈现；可切换为双列纵向密集浏览模式，帮助查找与回顾。桌面端在保持同一阅读逻辑下扩展为合理多列布局。
- 房间使用两层固定顶部结构：房间名称栏下方常驻 Chat、Board、Itinerary 三项按钮导航；页面仅由按钮切换，移动端不使用左右滑动切换。
- 聊天采用克制的弱气泡/排版分组，而非强 iMessage 式彩色大气泡。
- 房间顶部名称栏左侧放返回 Rooms 的图标，中间房间名、状态与倒计时按整个屏幕严格居中，右侧放必要操作；不显示当前页面指示图标。
- 倒计时平时低调，最后 15 分钟提升提示强度，最后一分钟使用轻微、不持续的动态提醒。
- 创建房间采用逐步引导而非长表单；第一层仅包含名称、模式、时长、人数和进入方式，其余设置收纳于 `More settings`。
- 无 Host 房间以克制的 `Community-led` 模式徽章明确标识。
- 邀请链接页采用活动邀请卡，先展示基本活动信息再要求昵称与头像；审批等待页保留公开的基本信息与柔和等待反馈，但绝不泄露房间内容。
- 聊天中同一成员连续消息仅在首条显示头像与昵称；系统消息居中、小字号并用极弱分隔；输入区固定底部，发送与按住说话分列；置顶消息以可收起的顶部窄卡呈现。
- 画板从中心空白区域开始，内容自然散布；选中内容后显示浮动工具栏，上传按钮可常驻右下角；顺序显示使用沉浸式全屏照片流。
- 进入进行中房间默认打开 Chat，之后记住成员上次使用的房间内页面与各页滚动位置；三页保持挂载，仅通过顶部导航按钮切换，并使用短促、克制的透明度过渡。
- 创建向导的选项使用大号可点击卡片，显示低干扰的步骤进度（如 `2 / 5` 与细进度线），不使用传统表单下拉框。
- 全局移动端保持顶部导航与悬浮创建按钮，不引入社交应用式五栏底部导航。
- 桌面端聊天右侧信息栏默认显示简介、倒计时、成员、置顶消息与行程摘要，并可在窄屏收起。

## 5. 动效与性能约束

- 动效采用轻快、弹性、清晰的 iOS 式反馈；页面切换与归档处理保持更安静、克制。
- 动效必须服务于操作确认、层级变化和空间感，不能只是装饰。
- 在支持 120Hz 的设备上尽可能接近 120fps；一般设备稳定流畅为最低要求。优先保证输入、滚动、拖动和画板手势。
- 避免高斯模糊、大面积滤镜、持续复杂阴影动画和会触发主线程重排的动画。优先使用 `transform`、`opacity`，并支持系统“减少动态效果”。

## 5.1 画板可用空间边界

- 为避免照片/注释被放到远离主体内容的“离群区域”，画板并非无限自由平面。
- 用户拖放到过远位置时，系统无打断提示地自动吸附至最近合法位置。

## 5.2 已确认：画板边界、状态与归档

- 采用有限但宽松的固定活动画布：以中心为起点，约为手机可视区域四倍宽、三倍高；所有内容必须留在其中。越界拖放无提示地自动吸附至最近合法位置。
- 数量标记和普通状态使用低饱和灰阶与形状区分；只在风险或临近结束时克制使用暖色，不使用醒目的红色大徽章。
- 房间结束后进入安静的 `moment is being preserved` 归档处理页，展示只读进度与数据正在保存的反馈。
- 归档详情采取“活动回忆册”结构：先显示大幅画板预览和活动信息，再进入 Chat、Board、Itinerary；归档卡片默认展示完整画板，可由每位成员单独选择自己归档列表中的照片封面。
- 空状态以情绪化短句为主，并保留一行明确行动提示。

## 5.3 品牌与可访问性交互

- 第一版只使用字重偏重的文字字标 `EventSpace`，暂不设计独立 Logo 图形；图钉只作为产品内的视觉元素。
- 图标采用细线、圆角、几何一致的体系，选中时以填充或加粗区分，不使用彩色图标。
- 当前 Phase 0 不纳入无障碍设计目标；设计和实现重心为主视觉、信息层级、操作流畅度与 Chromium/Safari 的性能。无障碍需求留待产品主体验稳定后另行评估，不作为当前页面的视觉约束。

## 5.4 地点展示

- 第一版行程卡片只展示地点文字与跳转 Google Maps / Apple Maps 的外链图标，不嵌入或预览真实地图。
- 原因：真实位置缩略图需要地图服务 API、密钥与可能的计费；当前优先避免这项额外依赖与成本。

## 6. 首页、卡片与回顾浏览

- 首页以“进行中”和“已归档”两个同级标签切换，默认显示进行中房间。
- 公开 Landing 仅用于未登录展示，不与登录后的 Rooms/Room 处于同一产品导航层级；登录后的 Rooms 取消营销标题，首屏直接显示状态切换与房间卡片。
- 默认浏览方式为单张大卡片的横向滑动流，营造活动杂志的阅读感；搜索、筛选和收藏仅改变结果集，不改变单张横滑的默认表现。
- 用户没有房间时，只显示一句情绪化短句和 `Create a room` 按钮，不放插画或大面积说明。
- 搜索与筛选合并为顶部紧凑搜索框，展开后提供名称搜索、状态与收藏筛选。收藏采用卡片右上角极小书签图标。
- 移动端创建按钮是带加号的圆角胶囊；滚动时可收为仅加号图标。账户入口只用右上角头像，菜单包含主题、账户、已删除归档与退出。
- 进行中房间在信息区以细状态点和剩余时间呈现，不把倒计时覆盖在画板缩略图上。
- 邀请页仅展示名称、活动说明、结束时间和人数上限，不泄露聊天、画板预览或完整成员名单。

## 7. 画板详情、顺序流与行程/投票卡

- 点击照片进入纯色遮罩的沉浸式详情层，包含大图、发布者、时间、注释、评论和表情；不使用毛玻璃。
- 移动端评论从底部上推，桌面端位于大图右侧窄栏。
- 自由画板只提供手势缩放和“回到全览”按钮，不常驻百分比或加减控件；空画板在中央显示极淡的邀请语与上传入口，首个内容出现后消失。
- 顺序显示采用两列纵向照片流，按上传顺序排列；点击进入详情，顶部固定 `Back to board`。
- 行程按时间纵向排列，在跨日处加入简洁日期分隔。
- 投票结束后的聊天卡长期保留，但仅展示问题、最终票数、结果与结束时间，不列出个人选票详情。

## 8. 成员、管理与高风险状态

- 成员列表按 Host、管理员、普通成员排序，每人只显示头像、昵称与必要的角色标签；不显示行程参与状态，避免列表拥挤。
- 行程负责人在行程卡上显示圆形头像与昵称，点击打开简短成员资料面板。
- 审核申请使用明确的“通过 / 拒绝”按钮，不使用滑动手势。
- 无 Host 房间的投票卡明确标为 `Community vote`，并显示达成过半数通过的时刻。
- 举报入口仅在消息、照片和成员的更多菜单中出现，不做常驻警示按钮。
- 禁言成员保留输入区域但显示简短状态说明；被踢出或拉黑成员只看到访问终止页。
- 房间临近冻结时，三页顶部使用同一条细提示；不弹出遮挡式倒计时。
- 删除个人归档前，明确说明“只会从你的列表移除，不影响其他成员”，并需要二次确认。

## 9. 身份、分享、主题与付费呈现

- 登录/补全身份使用全屏极简认证页，并显示关联房间名称作为上下文；不使用小型登录弹窗。
- 分享使用底部分享卡：二维码为主，附复制链接、刷新二维码与作废邀请码等操作。
- 付费不设置常驻入口；创建房间时可选择，或仅在成员、照片、容量、时长或归档需求超出配额时解释升级选择。
- 深色模式不压暗真实照片与白色图钉条；媒体保留本身亮度与色彩。
- 主题切换约 200ms 平滑过渡颜色，不让页面内容重新入场。
- PWA 在首次访问显示一次轻量安装提示；真实系统安装请求仍遵循各浏览器许可与可用性。

## 10. Phase 0 输出清单

1. 基于本文绘制公开首页、创建向导、邀请/审批、进行中房间、归档回忆册、认证、异常与归档处理的线框图。
2. 形成明亮/深色 token：色板、字体选择、字号层级、间距、圆角、边框、阴影与动效曲线；不得使用渐变、毛玻璃或纹理。
3. 验证移动端单卡横滑、双列浏览、房间三页按钮切换、画板手势与桌面端扩展排版。
4. 以真实浏览器中的轻量原型检验滚动、切换与拖动性能，再进入功能开发。

## 11. 首屏、色彩与阴影细则

- 公开首页首屏使用全视口高度；抽象图钉照片和 UI 卡片构成背景层，情绪化标题与双按钮置于前景。不得使用渐变或毛玻璃。
- 首屏进入时，背景照片与 UI 卡片分时轻微上移后定格；不做持续漂浮动画。
- 整体绝对主色为黑、白与暖灰；只在临近结束或风险状态使用极少量低饱和暖色，不引入独立品牌强调色。
- 主按钮在明亮模式为墨黑实底和浅色文字，在深色模式反转为暖白实底与深色文字。
- 卡片使用静态、柔和、多层阴影；悬浮与按下只改变 1–2px 位移和阴影强度，不动画阴影模糊半径。

## 12. 已确认：字体性能方案

- 字体不使用浏览器运行时向第三方字体站请求的形式。
- 保留一套极轻的无衬线界面字体，并只为公开首页、关键标题和归档标题加载一套经 Latin 子集化的衬线展示字体；聊天、表单和密集界面不加载或不强制使用展示衬线。
- 字体文件需有独立性能预算，并在首屏可先使用指标匹配的系统回退字体；展示字体加载完成后不应造成可见布局跳动。
- 具体字体家族、实际字重和文件体积将在视觉稿制作时按本方案验证；若达不到性能预算，优先缩减字重/子集或更换展示字体。

## 6. 尚待确认

1. 公开首页的具体画面编排与首屏动效。
2. 登录后首页卡片滑动/双列切换的控制样式与桌面网格细节。
3. 创建房间、加入房间、审批、归档、支付等流程的页面布局。
4. Chat、Board、Itinerary 三页的细部信息层级、工具栏、空状态和异常状态。
5. 字体、图标系统、品牌字标的最终选择及授权。
