# EventSpace 产品需求基线（进行中）

> 状态：已根据当前讨论整理。本文只记录已确认的产品决定与明确待定事项；未确认内容不应视为实现需求。  
> 最后更新：2026-07-13

## 1. 产品定位

EventSpace 是一个面向派对、酒吧、社区活动、婚礼等活动场景的私密临时协作空间。每个房间以实时聊天、自由画板与独立行程页为核心；到期或被结束后自动冻结为只读归档。

产品强调：

- 移动端优先，同时适配桌面端。
- 兼容 Chromium 与 Safari/WebKit。
- 极简、高级的黑白视觉；同时支持明亮和深色模式。
- 使用圆角、卡片与阴影表达层级。
- 操作要有克制、流畅、接近 iOS 体验的动效；性能优先，避免妨碍操作的动画。
- 视觉定稿补充：强沉浸活动感、情绪化首屏文案、暖灰/墨黑而非硬黑白、杂志式衬线标题与无衬线正文；全站不使用渐变、拟物纸张/木纹或高斯模糊。
- 数据安全按默认安全设计；第一版采用托管加密，不使用严格端到端加密（E2EE）。

## 2. 房间模型

### 2.1 可见性与进入方式

- 房间默认不公开，只能受邀进入。
- Host/创建者可设置进入方式：链接、二维码或邀请码。
- 二维码是房间链接的可视化形式。
- 邀请码默认无有效期；Host 可手动作废邀请码，或生成新的二维码/链接。
- 创建者可选择链接访问策略：持链接直接加入、持链接仍需邀请码，或持链接必须经审核。
- 地理位置限制暂不纳入第一阶段。
- 创建者可选择是否需要入房审核。
- Host 模式需要审核时，Host/管理员可看到申请者的头像、昵称与申请备注；创建者选择此模式即应实时处理申请。
- 无 Host 模式需要审核时，每个申请自动形成仅现有成员可见的入房投票，并沿用“超过当前有效成员半数”规则；不设置自动放行策略。
- 无 Host 房间的成员名单固定对房间内所有成员可见，不提供“仅 Host/管理员可见”这一无有效受众的选项。

### 2.2 生命周期

- 创建房间时设置房间时长；Host 模式下 Host 可在运行中修改时长，也可手动结束房间。时长最短为 15 分钟；免费房间最长 24 小时，付费房间最长 7 天；单次最多延长 24 小时，且不能缩短至当前时刻之前。
- 无 Host 房间的投票延长遵循同一套餐累计时长上限：免费房间累计最多 24 小时，付费房间累计最多 7 天，不允许借连续投票无限延长。
- 到期使用服务器时间为唯一准则，届时立即拒绝所有写操作并冻结房间，不设缓冲期。
- 房间可由投票结束；无 Host 模式下，结束房间须获超过半数票。
- 房间结束后先显示“正在归档”；通常数秒内完成，媒体较多时转为后台处理，并在归档就绪后通知成员。
- 房间结束前默认在 1 小时、15 分钟和 1 分钟提醒成员；Host 可关闭前两次提醒，但不可关闭最后一次提醒。
- 冻结后，聊天、行程与画板均可浏览，但都不可再修改或操作。
- 归档中的照片可以下载；聊天室、行程和画板历史均可浏览。

### 2.3 归档

- 归档仅对该房间未被踢出的原成员开放，且查看归档必须登录。
- 归档采用“个人归档副本”模型：每位已登录成员拥有自己的只读归档入口。
- 访客若在房间结束后才登录，只要当前设备仍保留该房间的临时身份，即可将个人归档绑定到该账户。
- 归档后，Host、管理员及普通成员均只能从自己的归档列表移除该房间，不能删除其他成员的归档。
- 免费归档保留 30 天；永久归档按房间购买的永久权益保留。到期、合法删除请求及其备份清理按已定义的数据删除周期执行。
- 免费归档到期后，主存储数据在 7 天内删除，备份在额外 30 天内清理；到期前 7 天提醒成员。
- 用户不导出完整房间数据；允许保存房间内照片等媒体。
- 用户删除账户后，显示名变为“Deleted member”、头像移除；内容保留至房间归档到期，永久归档中的内容保留但不再关联该账户。

## 3. 角色与房间模式

### 3.1 Host 模式

- 创建者为 Host。
- Host 可转让身份、添加管理员并撤销管理员。
- Host 仅可向房间内已登录成员转让身份，接收者明确确认后生效；原 Host 自动转为管理员。
- 管理员数量第一版不设固定上限，添加时向 Host 显示明确的权限风险提示。
- 管理员拥有几乎所有 Host 权限，但不能延长/缩短房间时长，也不能结束房间。
- Host 和管理员可置顶消息、删除其他人的消息、删除其他人的画板内容、禁言、踢人及处理入房申请。
- Host 和管理员可以发起无 Host 模式中允许的各类投票。

### 3.2 无 Host 模式

- 房间仍由创建者创建；创建时长、入口方式等初始设置均由创建者配置。
- 创建完成后，创建者即为普通成员，没有 Host 权限。
- 房间内决策由投票产生；普通成员可发起可用投票。
- 任一需要投票的事项，在获得超过半数票后生效。票数分母为当前已加入且未被踢出的全部成员；每项投票均应设截止时间。
- 两种房间模式在创建后不得互相切换。

### 3.3 角色失联

- 若 Host 或管理员离开/失联，不自动转让权限。
- 成员若要继续活动，可自行创建新房间并在旧房间发送链接。

## 4. 身份与登录

- 进入实时房间不强制登录，但必须设置头像和昵称。
- 登录方式：无密码邮箱一次性链接或验证码（magic link / OTP）与 Google 登录。
- 创建房间必须登录，以保障创建者的权限管理、付费能力与个人归档访问。
- 未登录访客可进行基础聊天功能：发送文字与短语音。
- 登录是下列操作的前提：上传画板照片、撰写画板注释、参加投票、查看归档。
- Host/管理员可以把未登录访客指定为行程负责人。
- 访客登录后，必须无缝继承此前的昵称、头像、聊天发言、投票和行程参与记录。
- 用户可修改头像与名称；改名时聊天中须显示“某某更改了名称”的系统消息。
- 访客头像可上传，也可选择系统生成的抽象头像或昵称首字母；同一房间内昵称必须唯一。
- 成员可主动离开房间；只要未被拉黑且当前入口策略允许，可通过原链接再次进入。需要审核的房间再次进入时须重新申请。
- 未登录访客关闭浏览器、清除本地数据或换设备后，临时身份失效；再次进入需重新设置头像和昵称。登录用户可跨设备恢复身份。
- 成员被踢出、拉黑或主动离开后，立即失去未来内容和实时房间的访问权；对其已查看、已复制或已下载的内容无法远程撤销。

## 5. 聊天

- 第一阶段聊天内容以文字为主；图片统一发布在画板，而非聊天区。
- 支持：文字、@ 提及、回复、消息撤回、消息置顶与按住说话的短语音消息。
- 单条短语音最长 60 秒，支持进度条拖动；不支持语音转文字或倍速播放。
- 归档后短语音可播放，但不提供下载按钮；这不构成对用户通过浏览器自行保存的绝对限制。
- 置顶仅限 Host 或管理员操作。
- 消息可在发出后两分钟内撤回，撤回后彻底不留痕。
- Host/管理员可删除其他成员消息。
- Host/管理员删除他人消息时不保留“已移除”占位记录。
- 消息不支持编辑，仅支持发出后两分钟内撤回；聊天输入遵循 Enter 发送、Shift+Enter 换行。移动端键盘 Enter 同样发送，换行通过输入框的明确换行操作完成。
- 消息不支持编辑，仅支持发出后两分钟内撤回。
- 不显示在线成员、输入中状态或消息已读回执。
- 语音消息默认绝不自动播放；用户点击后才播放，收到新语音仅显示静默提示。
- 聊天记录可搜索，但仅限用户所在的当前房间；搜索实现须与最终加密架构一致。
- 房间名称最长 80 字符、活动说明最长 500、聊天与评论最长 2,000、画板注释最长 500；均支持 Unicode 与 emoji。
- 需设计西方用户习惯下的系统消息、@ 提醒、置顶和搜索体验。
- 系统消息记录成员加入/离开、Host 转让、管理员变更、行程创建与状态变化、投票发起与结果、房间即将结束；不记录删除内容与举报细节。

## 6. 画板

- 画板是与聊天并列的核心区域，支持活动照片与简单文字注释的自由摆放。
- 成员可浏览画板；上传照片或撰写注释必须先登录。
- 所有已登录成员均可编辑自己发布的画板内容。
- 照片及其关联内容仅由发布者移动、缩放、旋转、编辑或删除；Host/管理员仅可删除他人内容。因而不需要处理多人同时移动同一照片的冲突。
- 画板内容归发布者本人所有；Host/管理员仅可删除他人内容，不可编辑。
- Host/管理员删除画板内容时不保留“已移除”占位记录。
- 每张照片在首次贴入画板时使用固定的微小随机倾斜角度，营造照片被贴在实体画板上的感觉；该角度在重新打开或重渲染时不得变化。
- 每张照片顶部延伸一条小白条，白条中央展示图钉视觉元素，表达“照片被图钉钉在画板上”。
- 支持直接调用移动设备相机拍照，以及从相册选择照片；桌面端支持拖拽上传。
- 新图片不得覆盖既有图片面积超过 20%。
- 标注不可被覆盖，文字不可压住图片主体。
- 需要防遮挡检测；发生冲突时不提示，拖动结束后自动吸附到最近的可用位置，不允许强行覆盖。
- 提供“顺序显示”模式：按上传时间排序展示照片及其注释。
- 顺序显示第一阶段只提供严格上传时间流与点击查看详情，不做日期分组或“仅照片/仅注释”筛选。
- 画板内容支持评论和表情回应；两者均必须登录。
- 画板评论支持短文本、回复、@ 提及、表情回应和两分钟内撤回，规则与聊天一致。
- 支持 iOS 与 Android 常见照片格式，包含 Live Photo/Motion Photo；原始文件最大 10 MB，上传后压缩并清除隐私元数据（如 EXIF）。
- 首次上传照片时，以简洁提示告知用户仅可分享自己拥有或已获授权分享的内容；不为每次上传增加勾选步骤。
- 浏览器不兼容动态照片时，将转换为静态主图，并向用户说明。
- 文字、评论和表情不设单独总量上限，仅受合理反垃圾限制；短语音计入房间总媒体容量。
- 免费房间画板总媒体容量为 250 MB；付费活动扩展包为 1 GB，且和照片数量上限同时生效。
- 图片压缩后最长边为 2560px、质量约 85，保留原始比例但不保存原图。

## 7. 行程

- 行程是独立页面，不强行集成于聊天区；聊天可用于讨论与决定行程。
- 每个行程展示：起止时间、地点、活动说明、地图和负责人。
- 地点支持地图搜索与手动文字输入；地图搜索优先，找不到时允许输入纯文字地点。
- 地图第一阶段使用地点卡片，并支持跳转 Google Maps / Apple Maps；不做路线或实时位置共享。
- 行程负责人由 Host 或管理员在设置行程时指定；负责人、Host、管理员可编辑或删除行程。
- 行程状态限定为未开始、进行中、已结束。定时模式由起止时间和权威当前时间推导；手动结束模式只能由负责人、Host 或管理员发出结束命令，实际结束时间由服务端写入，不允许客户端任意推进或回退状态。
- 第一阶段不提供行程报名、签到、容量或参与名单；页面专注于共享活动时间线。
- 浏览器推送是主要提醒方式。
- 创建者设置活动时区，默认其设备时区；所有行程以 UTC 保存，成员同时看到自己本地时间与活动时区时间。
- 夏令时遵循活动时区当地规则自动调整，不要求创建者手动计算 UTC 偏移。
- Host 模式下，仅 Host 与管理员可以创建行程和指定负责人。无 Host 模式下，所有行程相关提案均须先经投票生效。
- 负责人可为未登录访客，使用“临时访客身份 + 当前设备”绑定。

## 8. 投票

- 无 Host 模式中，房间决策均通过投票执行。
- Host 模式下，Host/管理员亦可发起无 Host 模式下的投票类型。
- 已明确投票可用于：行程安排、踢人、延长或缩短房间、共同决定负责人，以及结束房间；不提供画板内容投票。
- 非登录用户不能参与投票。
- 无 Host 房间中，所有已登录成员均可发起治理类投票；同一成员每 10 分钟最多发起一次此类投票。
- 无 Host 模式下，可投票事项暂定为：结束房间、延长/缩短房间、修改入口设置、踢人、禁言、变更行程负责人及行程安排。
- 无 Host 模式修改房间时长必须受安全上限约束；最小/最大时长、单次延长和总时长上限仍待确定。
- 第一阶段仅提供两类投票：赞成/反对的事项投票，以及多个行程方案中的单选投票；不提供多选或排序投票。
- 投票公开性由创建者设置为“完全公开”或“完全匿名”，但票数始终对成员显示，默认完全公开。
- 投票所需成员总数在发起时快照固定，防止成员中途加入或离开操纵结果。
- 当某一结果已超过所需门槛、结果确定时，投票立即生效，无需等待截止时间。
- 发起人可手动设置投票截止时间，但不得晚于房间结束时间。

## 9. 治理与安全

- 每位用户都可举报；举报由 Host 处理。
- 举报界面保持简单：举报者私密描述被举报者、违规发言或内容所在位置；Host 私密回复举报者，不强制公开或证明实际处理结果。
- 支持禁言、踢人和拉黑；禁言/踢人由 Host 或管理员决定。
- 禁言者仍可浏览房间、参与行程与投票，但不能发送文字、语音、@ 提及、画板评论、表情回应、照片或画板注释。
- 踢出仅将成员移出当前房间；若未被拉黑且入口仍有效，对方可重新申请加入。拉黑则禁止其再次进入该房间。
- 拉黑仅针对当前房间，并阻止同一用户再次进入。
- 将以隐私友好的设备标识辅助防止被拉黑者反复进入；该措施不能绝对阻止更换设备后的再次加入，此限制边界已接受。
- 设备级限制使用保存在浏览器中的随机设备令牌，而不做浏览器指纹追踪；用户清除网站数据后令牌失效，此限制边界已接受。
- 邀请码作废、链接轮换、入房审核是设备限制的补充手段。
- 踢人投票或管理员踢人一旦生效，立即终止该成员实时连接、停止未来内容访问并撤销其邀请码访问。
- 举报不自动转交原始内容，而由举报者在私密举报中详细描述违规者及内容位置；Host 依据描述处理。
- 所有写操作、权限校验、冻结校验均必须以服务端授权和服务端时间为准，不能信任客户端。
- 允许收集最小化、匿名化的自有运行指标（如崩溃率、性能错误、房间创建量和功能使用次数），不收集聊天、语音、照片或画板内容作为分析数据。
- 对访客实施最小反垃圾限制：限制每分钟文字、@ 提及和语音发送次数；异常时要求简单人机验证。

## 10. 加密与隐私架构

- 第一版采用托管加密，不使用严格 E2EE；因此用户可通过正常登录跨设备访问个人归档，无需保管恢复密钥。
- 必须落实传输加密、静态加密、最小权限、服务端权限校验、私有媒体存储、短期签名访问、访问审计和不含内容的日志。
- 隐私政策必须准确说明服务提供者可在运营、安全、故障处理和依法合规的必要范围内处理内容，不得将服务表述为 E2EE。
- 不出售用户数据、不投放广告、不做第三方行为追踪；员工不应常规人工查看房间内容，例外访问必须受最小权限与审计流程约束。
- 无论选择何种架构，推送通知均不应包含敏感内容正文，并采用隐私安全的提醒文案。
- 未登录访客仅收到当前打开页面内的提醒；已登录用户才能选择开启离线浏览器推送。
- 浏览器通知权限绝不在首次打开时请求，仅在用户主动点击“开启提醒”后请求系统授权。

## 11. 商业化与容量

- 不提供个人月度订阅；所有付费均为房间级付费，解锁权益附着于房间并对所有符合资格的成员生效。
- 免费房间会有可贴照片数量上限。
- 免费方案最多 10 名成员、25 张照片、250 MB 媒体容量、30 天归档保存期。付费能力、价格及未来商业数值保留配置化调整空间。
- 房间人数上限由创建者设定；典型房间为数人，最大规模为数十人。
- 房间可在创建时付费，或在运行中、免费归档未过期前升级。第一版由任一已登录成员单独支付，不提供站内分摊；成员分摊付款仅保留为未来可扩展能力。免费归档一旦过期并删除则不可付费恢复。
- 永久归档是独立的一次性房间级付费选项，和更高成员数、更多照片、延长房间时长等活动扩展权益分开收费。其权益附着于房间并惠及所有符合归档资格的成员。
- 永久归档允许日后另购容量扩展包，但扩展仅增加容量，不改变已归档内容或房间成员。
- 初始付费活动扩展包暂定为最多 50 名成员、200 张照片、7 天房间时长、1 GB 压缩后媒体容量；该商业化数值必须保持可配置，允许后续调整。
- 永久归档沿用房间购买时的容量上限，并透过高质量压缩降低长期存储成本。
- 永久归档生成并生效后不退款；支付失败不解锁权益。付款人删除账户不影响其他原成员继续访问该房间。
- 支付第一阶段应使用 Stripe 托管信用卡支付与基础税务能力，服务不自行保存支付卡资料。

## 12. 首页与房间创建

- 首页按“进行中”与“已归档”组织用户所属房间。
- 房间卡片展示画板缩略图、房间名称、状态、结束时间与成员数。
- 首页支持收藏、搜索与筛选。
- 创建房间需填写：房间名称、可选活动说明、模式（Host/无 Host）、时长、人数上限、进入方式、是否审核、成员名单/投票可见性。
- 创建房间可选填写封面图与地点字段。

## 13. 交互与成员资料

- 移动端画板使用以下手势：单指拖动画布或元素、双指缩放画布、长按选中元素；选中后出现简洁工具条。
- 主题默认跟随系统，用户可在设置中固定为明亮或深色；不提供按活动场景选择主题。
- 动效与交互以支持 120Hz 的设备上尽可能接近 120fps 为目标，在一般设备上至少保持流畅的 60fps；当性能不足时优先保证输入、拖动和可读性，并遵循系统“减少动态效果”设置。
- 当前优先完成主视觉、信息层级与交互流畅度；无障碍设计要求不纳入第一阶段的视觉定稿，待核心体验稳定后单独评估。
- 用户头像详情仅显示头像、昵称、房间角色和行程参与状态；不提供全局个人主页、粉丝关系或陌生人私信。
- 第一阶段通知覆盖：@ 提及、回复、被指定为负责人、行程状态变更、投票发起/即将结束/结果、入房申请、被批准加入、房间即将结束与归档完成。
- 房间内聊天、画板与行程使用顶部三项切换导航。桌面端保留三项明确入口，可按空间以主区与侧栏/独立页组合呈现，不强行同时塞满三块内容。
- 桌面端默认布局：聊天页为消息主区与右侧房间信息栏；画板页为全宽画板；行程页为主内容区与行程详情侧栏。
- 房间顶部固定栏显示房间名、倒计时、当前状态、分享入口、成员入口及更多操作，不放置付费/升级入口。
- 网络断线时保留未发送文字草稿，画板操作在重连后自动重试；若权限或房间状态已变化，则明确提示失败。
- 画板发布者可撤销/重做自己最近一次移动、缩放、旋转或文字编辑，不可撤销他人操作。
- 创建房间后提供干净空房间和可选的一句欢迎提示，不提供按活动场景的模板。
- 公开首页仅包含简洁产品说明、创建房间、通过链接进入、登录入口与隐私/条款链接；不展示公开房间或活动发现流。
- 首发正式支持当前及前一主要版本的 Chrome、Edge、Safari（macOS/iOS/iPadOS）；Firefox 不作为首发承诺浏览器，以集中保障 Chromium 与 Safari/WebKit 的动效质量。

## 14. 法律、隐私与地区

- 首发面向美国、欧盟与澳大利亚的英语用户，必须正确处理时区。
- 需要在开发前明确服务条款、隐私政策等法律与合规文档。
- 产品为通用活动协作工具，不按派对、婚礼、酒吧等使用场景划分年龄或提供主题选择。
- 统一最低年龄为 16 岁；第一阶段使用清晰的年龄资格声明，不收集出生日期。该门槛降低儿童数据处理风险，但服务条款、隐私政策和年龄设计仍须在上线前经目标法域律师审阅。
- 注册或创建房间时，用户必须勾选“我已满 16 岁并同意服务条款与隐私政策”；不要求填写生日。
- 账户仅收集邮箱或 Google 身份、显示名和头像；不要求手机号、出生日期或国家/地区。
- 不使用广告与第三方行为追踪，只保留安全、登录和必要功能 Cookie。
- 第一版提供支持邮箱或联系表单，以及公开的服务条款、隐私政策、社区规则和 Cookie 说明页面。
