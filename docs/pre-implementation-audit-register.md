# 开发前审计登记册

> 2026-07-14 首次只读审计。以下问题必须在对应功能进入开发前完成决策或在规格中闭合；本文件不改变既有已确认需求。

## P0：身份、权限与媒体安全

1. **访客升级登录的身份归属**：需要定义稳定的 `member/actor` 与认证账户的分离关系、当前设备证明、取消/失败路径、昵称冲突、事务合并与审计；历史消息不能靠改写作者 ID 认领。
2. **访客投票记录表述冲突**：既定规则是访客不能投票，但旧文本写“登录继承投票”。应改为继承聊天、昵称/头像、成员资格、负责人及行程参与记录，不包含投票。
3. **踢出后媒体的失效窗口**：60 秒签名 URL 无法在已签发后立即逐人撤销。需要接受“最坏仍可访问至 60 秒到期”，或改用授权代理读取/更短时效。
4. **上传隔离与转码链路**：浏览器压缩不足以满足服务端验证承诺。需要明确临时隔离 bucket、服务端验证/去 EXIF/转码、移入私有正式 bucket 与失败清理。
5. **Realtime 权威性**：客户端 Broadcast 只能做非权威临时同步；所有权限、踢出、投票、归档状态必须以数据库/RPC 结果为准，并定义踢出后的退订/重连行为。

## P1：生命周期、投票与商业边界

6. **归档资格快照**：需要定义主动离开成员、冻结瞬间访客、登录后认领、匿名账户清理与“共享只读快照 + 个人列表入口”的关系。
7. **房间状态机与竞态**：需要将 `active → freezing → archiving → archived → purge_pending → purged` 等状态、到期/手动结束/投票结束并发时的唯一获胜者、归档任务的重试明确为可验收规则。
8. **单选行程投票结算**：需要补齐过半规则、平票/无人过半、截止失效、行程变更或目标成员被踢后的处理。
9. **支付权益边界**：需要定义运行中/归档中/即将删除前的购买资格、支付时的容量与成员快照、失败/重复/乱序 webhook，以及永久归档与合法删除请求的用户可见结果。
10. **性能验收**：将“120Hz 尽可能 / 一般设备流畅”落为设备矩阵与指标，包括 iPhone Safari、Android Chromium、桌面 Safari/Chrome 的首屏、输入延迟、画板拖动和 10MB 上传测试门槛。

## P2：文档与视觉实现一致性

11. **创建封面图冲突**：旧文档中的“创建时可选封面图”与“全部图片在画板、默认完整画板为卡片封面”冲突。建议删除创建封面图字段；归档后仅允许每个成员为自己的列表选已有画板照片。
12. **顺序显示的稳定排序**：已确认两列纵向流，但必须保证上传时间可理解；不能使用按短列自动填充、导致阅读顺序混乱的瀑布算法。还需明确独立注释在顺序流中的位置。
13. **画板世界坐标**：视觉上的“手机视口四倍宽、三倍高”不足以跨设备同步。需要定义固定世界尺寸、缩放范围与元素边界，作为防遮挡/吸附的基准。
14. **导航术语**：统一使用 `Chat / Board / Itinerary`，不使用 `Plans`。
15. **地图边界**：Google Places 仅用于地点自动完成和外链；第一版不得渲染、缓存或展示地图瓦片/静态地图。
16. **归档处理文案**：非登录访客不保证后台通知；页面只承诺房间会在此处准备完成，已登录且允许推送者才可能收到通知。
17. **无障碍范围**：当前明确不作为 v1 页面设计/验收项；旧文档中的键盘、对比和减少动态效果要求需统一标注为未来评估，避免验收冲突。
18. **暂定商业/基础设施项**：USD、Google Places、`us-east-1`、Resend 等是当前建议，不应被旧文档同时标记为“未决”或被开发误认为上线前不可变更的商业决定。
